## Naming Authority

Treat meaningful names in the user-scoped Figma source as authoritative for:

- components and component sets
- variants and component properties
- semantic internal regions
- variables, collections, modes, and shared styles
- screens and flows

Preserve meaning, not invalid punctuation or casing. Do not rename a meaningful Figma concept merely because another name feels more familiar.

Ignore autogenerated structure names such as `Frame 123`, `Group 8`, `Rectangle 4`, `Instance 17`, or names that only describe coordinates. These are missing names, not contracts.

## Build a Naming Map

Create a compact map before implementation:

| Figma source | Code name | Kind | Decision |
| --- | --- | --- | --- |
| `Product card` | `ProductCard` | component and file | preserved, case-normalized |
| `State=Selected` | `state: "selected"` | variant prop | preserved, type-normalized |
| `Leading icon` | `leadingIcon` | prop or internal region | preserved, case-normalized |
| `Frame 1827` | `ProductMeta` | internal region | inferred from content and role |

Record inferred names that affect public props, exported components, files, tests, or reusable subcomponents. Tiny private variables need no ledger unless ambiguity could cause drift.

## Normalize Names for Code

Use these transformations without changing semantics:

- Figma components and component sets -> PascalCase React names
- React component files -> the exact PascalCase export plus `.tsx`
- properties and internal variables -> camelCase
- finite variant values -> stable lowercase string literals
- shared token names -> semantic kebab-case or the repository's established Tailwind variable convention
- acronyms -> the repository's existing acronym style

Map Figma variant axes directly when they form a coherent prop contract. For example, `Size=Large` and `State=Disabled` become typed `size` and `state` props rather than unrelated booleans.

## Infer Missing Names From Context

Infer a missing name in this order:

1. product function, such as `CheckoutSummary`
2. accessible or interaction role, such as `SearchInput` or `DismissButton`
3. rendered domain data, such as `ProductPrice` or `AccountStatus`
4. repeated structural role, such as `CardHeader` or `FilterGroup`
5. visual role only when no semantic signal exists

Choose the narrowest stable name that remains true across responsive layouts and states. Prefer `NotificationList` over `RightColumn`, and `ProfileAvatar` over `BlueCircle`.

Check sibling instances and flows before deciding. Repetition often reveals the intended component or property contract.

## Detect a Figma Design System

Treat the Figma file as containing a design system when it exposes one or more coherent sets of:

- variable collections with primitive or semantic naming
- modes such as light, dark, density, brand, or platform
- text, color, effect, or grid styles
- library components and component sets with controlled variants
- repeated semantic tokens used across several frames

Do not mistake a few repeated raw values for a complete design system. A design system has reusable naming or controlled contracts, not only visual similarity.

## Map the Design System to Tailwind

Build the token foundation before implementing product components:

1. Inventory Figma primitive values and semantic aliases.
2. Preserve primitive-to-semantic relationships instead of flattening every value.
3. Reuse equivalent repository tokens where they already match the Figma meaning.
4. Map remaining tokens into the Tailwind mechanism supported by the project, such as theme configuration, `@theme`, or CSS custom properties consumed by utilities.
5. Preserve modes with the repository's existing theme selector, media query, or data-attribute pattern.
6. Map Figma component-set properties to typed React variants.
7. Implement primitives and product components using the mapped semantic utilities.

Cover the Figma system categories present in the source, including color, typography, spacing, radius, border, shadow, sizing, breakpoints, and motion. Do not invent absent categories merely to fill a template.

Prefer semantic utilities such as surface, content, border, action, and feedback roles over copying raw palette names into every component.

## Fall Back to Tailwind Defaults

When the Figma file has no design system:

- use Tailwind's default color, spacing, typography, radius, shadow, sizing, and breakpoint scales as the baseline
- choose the closest default utilities that preserve hierarchy and layout intent
- add only the smallest evidence-backed theme extension needed for material fidelity
- keep true one-off measurements local
- do not present inferred aliases as if they came from Figma

Avoid creating a large custom token system from a single screen when Tailwind defaults already express the design coherently.

## Component File Contract

Place each reusable exported component in a matching named file:

```text
ProductCard/
  ProductCard.tsx
  index.ts
```

The primary export from `ProductCard.tsx` must be `ProductCard`. Follow an established flat-file repository convention when present, but still keep the export and filename identical.

Keep private internal regions in the same file when they are small and not independently reusable. Give each one a Figma-preserved or context-inferred semantic name. Extract it into its own matching file when it becomes reusable or owns a meaningful contract.

## Responsive Component Composition

Make one named component responsive when its responsibility and data contract remain stable across breakpoints:

- use Tailwind responsive utilities for layout, spacing, visibility, order, and typography
- keep fixed-format controls and media dimensions stable through explicit constraints
- expose props for meaningful design variants, not for viewport detection
- let the parent provide available space and route-level placement

Compose pages from those responsive components:

```tsx
export function SearchResultsPage() {
  return (
    <SearchResultsLayout>
      <SearchFilters />
      <ResultCollection />
    </SearchResultsLayout>
  );
}
```

Do not duplicate a desktop and mobile component inside the page solely because the arrangement changes. Split components only when the content contract or interaction model is genuinely different.

## Completion Checklist

- Every public component, file, prop, and variant name maps to meaningful Figma evidence or a defensible contextual inference.
- Autogenerated Figma layer names do not leak into the public code contract.
- A detected Figma design system is mapped to Tailwind-compatible tokens before product components use it.
- Tailwind defaults are used as the baseline when no Figma design system exists.
- Reusable exports live in matching component files.
- Responsive behavior lives in the component unless its contract materially changes.
- Pages import, compose, and connect responsive components instead of rebuilding their internals.
